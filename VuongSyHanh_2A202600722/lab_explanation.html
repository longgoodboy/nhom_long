1 Tổng Quan & Chủ Đề Dữ Liệu

Bài tập Ngày 8 nâng cấp hệ thống RAG lên một cấp độ thực tế (v2). Bạn sẽ tích hợp một pipeline Hybrid Search (Dense Retrieval + Lexical BM25) phối hợp với Reranking, và thiết lập một cơ chế fallback đặc biệt gọi là Vectorless RAG (PageIndex) phòng trường hợp tìm kiếm Hybrid không tìm thấy thông tin phù hợp.

Chủ đề Dữ liệu

Hệ thống được thiết kế để trả lời các câu hỏi dựa trên hai nhóm nguồn dữ liệu:
1. Pháp luật Việt Nam về ma tuý và các chất cấm (tập trung vào cấu trúc luật, hình phạt, danh mục).
2. Tin tức báo chí về các nghệ sĩ/người nổi tiếng liên quan tới ma tuý.

Hybrid Search

Kết hợp giữa sức mạnh tìm hiểu ngữ nghĩa (Semantic - Vector) và sự chính xác theo từ khóa (Lexical - BM25) để tối đa hóa độ phủ (Recall).

Vectorless Fallback

Nếu kết quả tìm kiếm Hybrid không đạt điểm tin cậy, hệ thống sẽ tự động fallback sang PageIndex SDK để tìm kiếm văn bản không cần qua vector DB nội bộ.

Context Reordering

Sắp xếp lại thứ tự các chunk trước khi đưa vào LLM để ngăn chặn hiện tượng "Lost in the middle" (LLM bỏ qua thông tin nằm giữa prompt).

Citation Generation

Bắt buộc LLM trích dẫn cụ thể tên tài liệu nguồn và năm ban hành (ví dụ: [Luật Phòng chống ma tuý, 2021]) trong câu trả lời.

3 Kiến Trúc Retrieval Pipeline Hoàn Chỉnh

Sơ đồ luồng xử lý câu truy vấn của người dùng từ lúc nhập câu hỏi đến khi LLM trả về câu trả lời cuối cùng:

User Query
"Hình phạt tàng trữ trái phép ma tuý?"
→
Hybrid Retrieve
Semantic Search + BM25 Lexical Search
→
Merge & Rerank
RRF/Score Fusion + Cross-Encoder Reranker
→
Decision Gate
Top Score ≥ 0.3?
No ➔
PageIndex Fallback
Vectorless Retrieval từ PageIndex AI Cloud
→
Context Reordering
Sắp xếp chunks tránh Lost in the middle
→
LLM Generation
Câu trả lời chính xác có Citation dạng [Luật, 2021]

4 Chi Tiết 10 Tasks Cá Nhân (50 Điểm)

Mỗi học viên cần tự tay hoàn thành các file mã nguồn từ task1 đến task10 nằm trong thư mục src/.

Task 1 — Thu Thập Văn Bản Pháp Luật 3 điểm

Mục tiêu: Thu thập tài liệu pháp lý định dạng PDF/DOCX liên quan tới ma tuý.

  • Tải tối thiểu 3 văn bản pháp luật (Luật phòng chống ma tuý, Nghị định 105/2021, Bộ luật Hình sự Chương XX, Nghị định 57/2022...).
  • Lưu file thô vào thư mục: data/landing/legal/
  • Đặt tên file rõ ràng, không dấu, không khoảng cách: ví dụ luat-phong-chong-ma-tuy-2021.pdf
  • Kích thước mỗi file phải đảm bảo lớn hơn 1KB.

Task 2 — Crawl Bài Báo 3 điểm

Mục tiêu: Crawl dữ liệu tin tức về các scandal ma tuý của nghệ sĩ.

  • Crawl tối thiểu 5 bài báo (khuyến nghị dùng thư viện Crawl4AI).
  • Lưu vào thư mục: data/landing/news/ dưới định dạng JSON, HTML, hoặc MD.
  • Kích thước mỗi file phải lớn hơn 500 bytes.
  • Nếu lưu dạng JSON, bắt buộc phải có key "url" chứa link gốc bài báo để phục vụ trích dẫn sau này.

Task 3 — Convert Sang Markdown 4 điểm

Mục tiêu: Chuẩn hóa định dạng toàn bộ tài liệu thô về Markdown để phục vụ chunking chính xác.

  • Sử dụng thư viện markitdown (của Microsoft) để đọc tài liệu trong data/landing/ và chuyển đổi.
  • Ghi kết quả vào thư mục: data/standardized/ tương ứng (giữ nguyên phân cấp thư mục legal/ và news/).
  • Mỗi file kết quả có định dạng .md, dung lượng phải lớn hơn 200 ký tự.

Task 4 — Chunking & Indexing 7 điểm

Mục tiêu: Phân mảnh các tài liệu markdown và lưu trữ vào Vector Database.

  • Thiết lập cấu hình CHUNK_SIZE và CHUNK_OVERLAP (overlap phải nhỏ hơn chunk size).
  • Sử dụng langchain-text-splitters (ví dụ: RecursiveCharacterTextSplitter hoặc MarkdownHeaderTextSplitter) để cắt text.
  • Đảm bảo các chunk sau khi cắt không vượt quá CHUNK_SIZE * 1.1 (ngưỡng dung sai tối đa 10%).
  • Sử dụng mô hình embedding (như sentence-transformers/all-MiniLM-L6-v2 hoặc BAAI/bge-m3) để index dữ liệu vào Vector Database (Weaviate được khuyến nghị, hoặc Chroma/FAISS).

Task 5 — Semantic Search Module 6 điểm

Mục tiêu: Triển khai hàm tìm kiếm ngữ nghĩa (Dense Vector Search) trên Vector DB.

Yêu cầu hàm semantic_search(query: str, top_k: int = 10) -> list[dict]:

# Kết quả trả về phải là một list các dict có định dạng:
[
    {
        "content": "Nội dung đoạn văn bản tìm được...",
        "score": 0.852,  # Điểm tương đồng tương đối (từ cao xuống thấp)
        "metadata": {"source": "luat-phong-chong-ma-tuy-2021.md", "type": "legal"}
    },
    ...
]

Task 6 — Lexical Search Module 6 điểm

Mục tiêu: Triển khai hàm tìm kiếm từ khóa (tần suất xuất hiện từ) sử dụng thuật toán BM25.

Yêu cầu hàm lexical_search(query: str, top_k: int = 10) -> list[dict]:

  • Sử dụng thư viện rank-bm25 hoặc BM25 built-in của Weaviate.
  • Kết quả trả về phải được sắp xếp theo score giảm dần.
  • Nếu có từ khóa trùng khớp trong văn bản, điểm score phải lớn hơn 0 (không chấp nhận điểm toàn bộ bằng 0).

Task 7 — Reranking Module 6 điểm

Mục tiêu: Đánh giá lại độ liên quan để chọn lọc các chunk tốt nhất đưa vào LLM.

Yêu cầu hàm rerank(query: str, candidates: list[dict], top_k: int = 5) -> list[dict]:

  • Sử dụng một trong các phương pháp: Cross-encoder (ví dụ: mô hình Jina Reranker v2 API hoặc Qwen Reranker local), MMR (Maximal Marginal Relevance) tự triển khai, hoặc RRF (Reciprocal Rank Fusion).
  • Kết quả sau khi rerank phải trả về đúng số lượng top_k, được cập nhật giá trị "score" mới và sắp xếp giảm dần.

Task 8 — PageIndex Vectorless RAG 4 điểm

Mục tiêu: Thiết lập cơ chế fallback tìm kiếm không cần vector DB nội bộ thông qua dịch vụ đám mây PageIndex.

Yêu cầu hàm pageindex_search(query: str, top_k: int = 5) -> list[dict]:

  • Đăng ký tài khoản trên pageindex.ai, lấy API Key và upload tài liệu.
  • Sử dụng SDK pageindex để thực hiện tìm kiếm.
  • Kết quả trả về từ hàm này bắt buộc phải có thuộc tính "source": "pageindex" trong phần metadata để test suite nhận diện.

Task 9 — Retrieval Pipeline Hoàn Chỉnh 7 điểm

Mục tiêu: Tích hợp tất cả các module tìm kiếm đơn lẻ thành một pipeline thống nhất có cơ chế kiểm tra điểm ngưỡng (threshold fallback).

Yêu cầu hàm retrieve(query: str, top_k: int = 5, score_threshold: float = 0.3) -> list[dict]:

1. Chạy song song/nối tiếp: semantic_search + lexical_search.
2. Hợp nhất (Merge) kết quả và tiến hành Rerank (Task 7).
3. Kiểm tra kết quả có score cao nhất:
   - Nếu score ≥ score_threshold: Trả về kết quả Hybrid (đánh dấu source="hybrid").
   - Nếu score < score_threshold: Gọi pageindex_search (đánh dấu source="pageindex").
4. Đảm bảo kết quả trả về không vượt quá top_k và không crash nếu một trong các luồng bị trống.

Task 10 — Generation Có Citation 4 điểm

Mục tiêu: Lắp ráp dữ liệu context, xử lý tối ưu vị trí context và gửi prompt tới LLM để sinh câu trả lời có nguồn trích dẫn.

  • Triển khai hàm reorder_for_llm(chunks: list[dict]) -> list[dict]: Sắp xếp các chunk tìm được theo quy luật Lost in the middle (đưa chunk quan trọng nhất lên đầu và cuối danh sách, chunk ít quan trọng hơn nằm ở giữa prompt).
  • Triển khai hàm format_context(chunks: list[dict]) -> str: Ghép các chunk lại kèm theo thông tin metadata nguồn.
  • Triển khai hàm generate_with_citation(query: str) -> dict: Trả về dict chứa key "answer" (câu trả lời của LLM). Câu trả lời phải sử dụng thông tin từ context và chèn trích dẫn dạng [Tên văn bản, Năm] ngay sau câu khẳng định. Nếu context không có đủ thông tin, LLM bắt buộc phải trả về: "I cannot verify this information".

5 Yêu Cầu Bài Tập Nhóm (30 Điểm)

Sau khi các thành viên hoàn thành bài tập cá nhân, cả nhóm sẽ ngồi lại tích hợp code để xây dựng 1 trong 2 sản phẩm dưới đây:

Lựa Chọn 1

Sản Phẩm RAG Chatbot

Xây dựng một chatbot trực quan để tra cứu pháp luật ma tuý và tin tức liên quan.

  • Giao diện chat thân thiện (dùng Streamlit, Gradio, hoặc Chainlit).
  • Có hiển thị câu trả lời kèm citation (Task 10).
  • Hỗ trợ bộ nhớ cuộc hội thoại để hỏi các câu follow-up (Conversation Memory).
  • Có khu vực hiển thị danh sách các tài liệu nguồn (Source Documents) đã dùng để trả lời.
Lựa Chọn 2

RAG Evaluation Pipeline

Xây dựng pipeline đánh giá chất lượng hệ thống RAG sử dụng các framework chuyên dụng.

  • Sử dụng DeepEval, RAGAS, hoặc TruLens.
  • Xây dựng bộ dữ liệu kiểm thử (Golden Dataset) gồm ít nhất 15 cặp Q&A.
  • Đo lường ít nhất 4 chỉ số cốt lõi: Faithfulness, Answer Relevance, Context Recall, và Context Precision.
  • Thực hiện đánh giá so sánh A/B giữa 2 cấu hình RAG khác nhau và viết báo cáo phân tích.

Yêu Cầu Chung Cho Bài Nhóm:

  • Phải tích hợp được mã nguồn từ bài cá nhân của các thành viên.
  • Mã nguồn phải được push lên repository chung của nhóm.
  • Có file group_project/README.md mô tả kiến trúc hệ thống, hướng dẫn chạy và bảng phân công công việc cụ thể cho từng thành viên.
